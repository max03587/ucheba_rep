# Часть 1: Техническое задание (ТЗ) — Мобильная версия

## Разработка мобильного мессенджера «ChatApp» (аналог Telegram)

### 1. Цель проекта

Разработать **нативные мобильные приложения** для Android и iOS, обеспечивающие безопасный обмен сообщениями и мультимедиа с мгновенной доставкой и низким потреблением трафика.

### 2. Функциональные требования (Mobile First)

#### 2.1. Регистрация и устройства

- **Авторизация:** Только по номеру телефона (SMS/звонок с кодом).
    
- **Мультиплатформенность:** Пользователь может войти в аккаунт с нескольких устройств (телефон, планшет), но история синхронизируется с облака (как в Telegram).
    
- **Управление сессиями:** Отображение списка активных сессий, возможность завершить любую из них удаленно.
    

#### 2.2. Мобильный интерфейс

- **Адаптивность:** Дизайн должен подстраиваться под разные размеры экранов (телефоны, складные устройства, планшеты).
    
- **Жесты (Gestures):** Свайп для ответа на сообщение, долгое нажатие для вызова меню действий, пинч-ту-зум для фото.
    
- **Интерактивные уведомления:** Возможность ответить на сообщение прямо из уведомления (Notification Actions) без открытия приложения.
    

#### 2.3. Работа с памятью и кэшем

- **Оффлайн режим:** Полный доступ к истории сообщений (кэш) при отсутствии интернета.
    
- **Автоматическая очистка кэша:** Настройка автоматического удаления старых медиафайлов с устройства (через 1 неделю/1 месяц) для экономии места.
    

#### 2.4. Медиа и звонки

- **Камера:** Встроенный фото/видео захват без выхода из приложения (In-app camera).
    
- **Видеозвонки:** Оптимизация под мобильные сети (адаптивный битрейт при переходе с Wi-Fi на 4G).
    
- **Голосовые сообщения:** Удобная запись одним нажатием с возможностью послушать перед отправкой.
    

### 3. Требования к мобильной безопасности

- **Биометрия:** Вход в приложение по FaceID / TouchID / сканеру отпечатков пальцев (локальный замок).
    
- **Шифрование БД на устройстве:** Локальная база данных сообщений (кэш) должна храниться в зашифрованном виде (SQLCipher или аналоги).

# Часть 2: Исследование технологий (Mobile Stack)

Для мобильного приложения у нас есть две основные "религии": **Нативная разработка** (Kotlin и Swift по отдельности) и **Кроссплатформенная** (React Native, Flutter).

### 1. Нативная разработка (Рекомендуемый путь)

Это путь Telegram, WhatsApp, Signal. Дороже, но дает максимальный контроль и производительность.

#### Android (Kotlin)

- **Язык:** **Kotlin** (современный, безопасный, рекомендован Google).
    
- **UI:** **Jetpack Compose** (современный инструментарий для построения интерфейсов).
    
- **Сеть:** **Retrofit** + **OkHttp** (для HTTP запросов) + **WebSockets** (для реалтайма).
    
- **База данных (локальная):** **Room** (надстройка над SQLite) для хранения кэша сообщений. В сочетании с **SQLCipher** для шифрования.
    
- **Фоновые задачи:** **WorkManager** (для гарантированной отправки сообщений при плохом интернете).
    
- **Push-уведомления:** **Firebase Cloud Messaging (FCM)**.
    

#### iOS (Swift)

- **Язык:** **Swift**.
    
- **UI:** **SwiftUI**.
    
- **Сеть:** **URLSession** или библиотека **Alamofire** + **Starscream** (WebSockets).
    
- **База данных (локальная):** **CoreData** или **Realm** (Realm проще и быстрее для NoSQL подхода). Шифрование включается флагом.
    
- **Push-уведомления:** **Apple Push Notification Service (APNS)**.
    

**Сложность:** Придется держать две команды разработчиков (или искать разработчиков под обе платформы) и писать логику дважды.

### 2. Кроссплатформенная разработка (Быстрый старт)

Позволяет писать один код для Android и iOS.

#### Сравнение: Flutter vs React Native

|Критерий|Flutter (Dart)|React Native (JavaScript/TypeScript)|
|---|---|---|
|**Производительность**|**Выше** (компилируется в нативный код (ARM), свой движок рендеринга Skia).|**Средняя** (мост между JS и нативными компонентами).|
|**UI Компоненты**|Свои "нарисованные" компоненты (Material You/Cupertino). Выглядят одинаково на всех платформах.|Нативные компоненты (превращаются в настоящие iOS UI элементы).|
|**Сообщество**|Растет быстро (Google).|Огромное (Meta). Много готовых библиотек.|
|**WebRTC (звонки)**|Есть библиотеки, но интеграция сложнее.|Больше готовых решений для интеграции с VoIP.|
|**Риски**|Приложения весят больше (~10-15 МБ минимум).|Зависимость от "моста" может лагать в сложных анимациях.|

### 3. Серверная часть (Backend) — остается неизменной

Независимо от выбора мобильной платформы, сервер должен быть мощным. Рекомендации из предыдущего ответа актуальны:

- **Backend:** **Golang**.
    
- **Базы данных:** PostgreSQL + Redis + ScyllaDB (Cassandra).
    
- **API:** **WebSockets** для сообщений (постоянное соединение) + **gRPC** для быстрой внутренней коммуникации.
    

### 4. Рекомендации по выбору мобильного стека

#### Сценарий А: Если есть бюджет и нужен идеальный продукт (как Telegram)

- **Стек:** **Нативная разработка** (Kotlin + Jetpack Compose для Android, Swift + SwiftUI для iOS).
    
- **Плюсы:** Максимальная оптимизация батареи, трафика, плавность анимаций, легче внедрять новые фичи ОС (например, Material You или iOS Widgets).
    
- **Минусы:** Высокая стоимость разработки и поддержки.
    

#### Сценарий Б: Если нужно быстро запустить MVP и сэкономить

- **Стек:** **Flutter**.
    
- **Плюсы:** Одна кодовая база, быстрая разработка интерфейсов.
    
- **Минусы:** Сложность интеграции некоторых нативных модулей (особенно WebRTC видеозвонков с аппаратным ускорением). Приложение может потреблять больше памяти.
    

### 5. Дополнительные мобильные фишки

- **VoIP (Звонки):** Для того чтобы звонки работали как в Telegram (даже когда приложение закрыто), на iOS нужно использовать **PushKit** (специальный тип пушей для звонков), а на Android — **CallScreening API** или высокоприоритетные FCM.
    
- **Media:** Для быстрой загрузки изображений из кэша использовать библиотеки **Coil** (Android) и **Kingfisher** (iOS).
    

### Итоговый вердикт

Если вы планируете **долгосрочный проект**, который будет жить годы и конкурировать с монстрами — выбирайте **Натив (Kotlin + Swift)**.  
Если вы **стартап** с ограниченным бюджетом и временем — выбирайте **Flutter**, но будьте готовы "допиливать" нативные модули для звонков и оптимизации батареи.
